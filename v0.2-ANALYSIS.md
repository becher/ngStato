# ngstato v0.2 — Deep Feature Analysis

## Executive Summary

**ngstato is surprisingly feature-complete for a 0.1 release.** Most foundational pieces are well-implemented. The gaps for v0.2 are not "missing core features" but rather "missing patterns and utilities that users expect from modern state managers."

---

## 1. Core Store Features (@ngstato/core - store.ts)

### ✅ IMPLEMENTED

| Feature | Status | Details |
|---------|--------|---------|
| **State Management** | ✅ Implemented | Immutable state with private `_state`, exposed via getters |
| **Actions (sync/async)** | ✅ Implemented | Dispatched via `dispatch()`, wrapped with Proxy for mutations |
| **Computed Values** | ✅ Implemented | Via `computed` config, memoized per call via `getComputed()` |
| **Lifecycle Hooks** | ✅ 6 hooks! | `onInit`, `onDestroy`, `onAction`, `onActionDone`, `onError`, `onStateChange` |
| **Subscription System** | ✅ Implemented | `subscribe()` returns unsubscribe function |
| **Error Handling** | ✅ Implemented | `onError` hook catches action errors, throws after logging |
| **Cleanup Registry** | ✅ Implemented | `registerCleanup()` for managing resources (used by fromStream) |
| **State Immutability** | ✅ Enforced | State is read-only from outside, read-only return from `getState()` |

### 🚫 NOT IMPLEMENTED

| Feature | Why Missing | Impact |
|---------|------------|--------|
| **Selectors** | Never needed - computed values serve the purpose | Low priority but useful for complex derivations |
| **Memoized Selectors** | Computed are re-evaluated every call | **Medium priority for v0.2** |
| **State Validation** | No schema validation | Dev-only feature, low priority |
| **Middleware Pipeline** | Hooks are sufficient for v0.1 | **Medium-high priority for v0.2** |

**Key Finding:** The 6 hooks system is actually quite powerful. It's a minimal version of middleware that avoids the complexity of redux-style enhancers.

## 2. HTTP Client Features (@ngstato/core - http.ts)

### ✅ IMPLEMENTED

| Feature | Status | Details |
|---------|--------|---------|
| **HTTP Methods** | ✅ All 5 | GET, POST, PUT, PATCH, DELETE |
| **Base URL** | ✅ | Configured via `StatoConfig.baseUrl` |
| **Query Parameters** | ✅ | Via `params` object in RequestOptions |
| **Auth Headers** | ✅ | Injected token via `StatoConfig.auth()` function |
| **Timeout** | ✅ | Configurable timeout with AbortController |
| **Error Class** | ✅ | `StatoHttpError` with status + body |
| **AbortSignal** | ✅ | Passed to fetch for cancellation (used by abortable) |
| **Global Instance** | ✅ | `http` singleton + `configureHttp()` |
| **Zero Dependencies** | ✅ | Pure `fetch` API, no HttpClient required |

### 🚫 NOT IMPLEMENTED

| Feature | Why Missing | Impact |
|---------|------------|--------|
| **Interceptors** | Could be middleware layer | **Medium priority for v0.2** |
| **Request/Response Transforms** | Keep it simple for v0.1 | Low priority |
| **Retry Logic** | Outsourced to `retryable()` helper | Already works via helpers |
| **Caching** | Outsourced to computed/actions | Low priority |

**Key Finding:** HTTP client is production-ready. The architecture elegantly delegates retry/abort to helper functions instead of baking it in.

## 3. Action Helper Functions (@ngstato/core/helpers)

### ✅ IMPLEMENTED (6 Helpers)

| Helper | Status | RxJS Equivalent | Works In | Example |
|--------|--------|------------------|----------|---------|
| **retryable()** | ✅ | `retryWhen` | Student store | `loadStudents: retryable(async fn, { attempts: 3, backoff: 'exponential' })` |
| **optimistic()** | ✅ | `optimisticUpdate` | Student store | `deleteStudent: optimistic(immediate, confirm)` |
| **abortable()** | ✅ | `switchMap` | Student store | `searchRemote: abortable(async fn)` cancels prior requests |
| **debounced()** | ✅ | `debounceTime` | Student store | `search: debounced(fn, 300)` waits 300ms |
| **throttled()** | ✅ | `throttleTime` | Student store | `scroll: throttled(fn, 100)` fires max every 100ms |
| **fromStream()** | ✅ | `from()` + `subscribe()` | Student store | `listenNotifications: fromStream(observable, updateFn)` |

### 🚫 NOT IMPLEMENTED

| Pattern | Why Missing | RxJS Equivalent | Priority |
|---------|------------|-----------------|----------|
| **DistinctUntilChanged** | Could be auto in setter? | `distinctUntilChanged()` | Low |
| **CombineLatest** | Multi-store coordination | `combineLatest()` | Medium - v0.2 |
| **Race/ForkJoin** | Parallel async patterns | `race()`, `forkJoin()` | Medium - v0.2 |

**Key Finding:** All the **essential** RxJS patterns are covered. This is a well-designed reduction of RxJS for a simpler state manager.

## 6. Reset & Hydration Features

### ✅ IMPLEMENTED (Manual Pattern)

```typescript
// In student-demo store:
reset(state: any) {
  state.students      = []
  state.selected      = null
  state.isLoading     = false
  state.error         = null
  state.searchQuery   = ''
  state.notifications = []
}
```

**This works but is:**
- ✅ Explicit (good for testing)
- ❌ Manual (requires typing every field)
- ❌ No built-in hydration from external storage

### 🚫 NOT IMPLEMENTED

| Feature | Why Missing | Impact |
|---------|------------|--------|
| **Auto-reset Utility** | Could auto-generate from initial config | Low priority |
| **localStorage Persist** | No built-in sync | **High priority for v0.2** |
| **Session Storage Hydrate** | No built-in restore | **High priority for v0.2** |
| **IndexedDB Support** | Complex, defer to users | Low priority |

**Key Finding:** Reset works well. Persistence/hydration is a **real gap** for v0.2.

## 7. Side-Effects / Effects Management

### ✅ IMPLEMENTED (Via Hooks)

```typescript
// Current pattern:
hooks: {
  onInit: (store) => {
    store.loadStudents()  // Auto-load on init
  },
  onStateChange: (prev, next) => {
    console.log('Changed:', prev, next)
  }
}
```

**This works but:**
- ✅ Simple (minimal API)
- ❌ Basic (no reactive dependencies)
- ❌ Can't express "run when X changes"

### 🚫 NOT IMPLEMENTED

| Feature | Why Missing | Impact |
|---------|------------|--------|
| **Reactive Effects** | "Run when X changes" pattern | **High priority for v0.2** |
| **Conditional Execution** | Effects with dependencies | **Medium-high priority for v0.2** |
| **Error Recovery** | Effects error handling | Medium - v0.2 |
| **Cleanup Functions** | Return cleanup from onInit | Medium - v0.2 |

**Key Finding:** Effects system is basically just hooks. A real effects API would enable:
```typescript
// What v0.2 should enable:
effect: {
  loadWhenQueryChanges: (state) => {
    return () => store.loadStudents(state.searchQuery)
  }
}
```

## Recommendations for v0.2 Release

### 🎯 **Priority 1: Foundation Features** (Launch Blockers)

1. **Built-in Selectors** (~2 hours)
   - Memoized selectors similar to Reselect
   - Signal-aware computed in Angular
   - ```typescript
     selectors: {
       filteredStudents: (state) => state.students.filter(...)  // auto-memoized
     }
     ```

2. **Persistence/Hydration** (~3 hours)
   - localStorage support via hooks
   - Automatic state serialization
   - ```typescript
     hooks: {
       onHydrate: async (store) => store._hydrate(),
       onPersist: (state) => localStorage.setItem(...)
     }
     ```

3. **Reactive Effects Pattern** (~2 hours)
   - Watch dependencies and auto-run side-effects
   - ```typescript
     effects: {
       loadWhenIdChanges: [
         (state) => state.selectedId,
         async (store, id) => store.loadStudent(id)
       ]
     }
     ```

### 🎯 **Priority 2: Polish Features** (Nice-to-have)

4. **Middleware Pipeline** (~2 hours)
   - Standardize hook system into middleware
   - Enable order-controlled execution

5. **Advanced Async Patterns** (~2 hours)
   - `combineLatest()` for multi-store coordination
   - `race()` for fastest-wins pattern
   - `forkJoin()` for parallel operations

6. **Testing Utilities** (~1 hour)
   - Test providers for Angular
   - State snapshot helpers

### ❓ **Not Recommended for v0.2**

- Time-travel debugging (too complex, DevTools sufficient)
- Undo/redo (niche use case)
- Schema validation (users can add via middleware)
- IndexedDB integration (localStorage is enough)

## Conclusions

### What's Surprising

1. **Action helpers are <100 lines each** — Very elegant patterns, no bloat
2. **No RxJS needed** — The dependency graph is minimal
3. **Hooks system is actually quite powerful** — Covers 80% of middleware needs
4. **Angular integration is perfect** — Zero friction with Signals

### Areas to Improve

1. **Persistence is completely absent** — Users will need this day-1
2. **Effects system is too basic** — Need reactive watch patterns
3. **Selectors concept missing** — Will cause derivation confusion in larger apps
4. **DevTools visual UI not started** — This is the polish gap

### Best Practices Observed

✅ Stores have factory functions, easily testable
✅ No global config hell (provideStato is clean)
✅ Angular lifecycle properly integrated
✅ HTTP client is zero-dependency
✅ Helpers are composable (abortable + debounced + retryable together)

### Architecture Strengths

- ✅ Clean separation: core/angular/helpers
- ✅ TypeScript support first-class
- ✅ No decorators needed (good friction point)
- ✅ SSR-safe (no DOM dependencies in core)
